\section{Organización general del proyecto}
\label{sec:organizacion-general-proyecto}

\subsection{Tecnologías utilizadas}

Las principales tecnologías utilizadas en la implementación del sistema son las siguientes:

\begin{itemize}
    \item \textbf{Python}: lenguaje principal del proyecto. Se ha utilizado por su integración con bibliotecas de inteligencia artificial, visión por computador e interfaces gráficas, así como por su flexibilidad para estructurar el sistema en paquetes independientes.

    \item \textbf{PySide6}: biblioteca utilizada para desarrollar la interfaz gráfica de escritorio. Forma parte del proyecto \textit{Qt for Python} y permite utilizar Qt6 desde aplicaciones Python \cite{pyside6_docs}.

    \item \textbf{OpenCV}: biblioteca empleada para la captura de cámara, el procesamiento de imágenes y la aplicación de transformaciones geométricas en el módulo de visión por computador \cite{opencv_docs}.

    \item \textbf{NumPy}: biblioteca utilizada como soporte para la representación y manipulación de matrices, imágenes y datos numéricos \cite{numpy_docs}.

    \item \textbf{Ultralytics/YOLO}: tecnología utilizada para integrar los modelos de reconocimiento visual, tanto en la detección de piezas como en la clasificación de cartas \cite{ultralytics_docs}.

    \item \textbf{pytest}: herramienta empleada para definir y ejecutar pruebas automatizadas sobre los principales módulos del sistema \cite{pytest_docs}.

    \item \textbf{Scripts propios}: conjunto de herramientas auxiliares desarrolladas para tareas de calibración, captura de datos, depuración visual, torneos entre agentes y medición del rendimiento de la búsqueda.
\end{itemize}

\subsection{Estructura del repositorio}

El código principal del proyecto se ubica en el paquete \texttt{src/onitama/}. La estructura general del repositorio es la siguiente:

\begin{verbatim}
TFG-Onitama/
├── src/onitama/
│   ├── engine/      # Motor de juego
│   ├── ai/          # Agente de inteligencia artificial
│   ├── vision/      # Visión por computador

│   ├── gui/         # Interfaz gráfica

\end{verbatim}

% 5.2 Motor de juego

\section{Motor de juego}
\label{sec:motor-juego}

El paquete \texttt{onitama.engine} concentra la implementación del estado de partida y de las reglas legales del juego. Sus módulos principales son \texttt{state.py}, \texttt{pieces.py}, \texttt{cards.py}, \texttt{moves.py} y \texttt{rules.py}.

\subsection{Estado y acciones}
\label{subsec:estado-acciones}

La representación central del motor es \texttt{GameState}. Se define como una \texttt{dataclass} inmutable mediante \texttt{@dataclass(frozen=True)}, de forma que cada posición de la partida se trata como un valor independiente. Esto resulta especialmente útil para la búsqueda de la inteligencia artificial, donde se generan muchos estados sucesores y es importante evitar modificaciones accidentales sobre estados ya explorados.

Un \texttt{GameState} almacena la información necesaria para reconstruir una posición completa:

\begin{itemize}
    \item \texttt{board}: tablero de 5 x 5, representado como una tupla de tuplas.
    \item \texttt{to\_move}: jugador al que corresponde mover.
    \item \texttt{red\_cards} y \texttt{blue\_cards}: cartas activas de cada jugador.
    \item \texttt{side\_card}: carta lateral disponible para el intercambio.
    \item \texttt{red\_master\_pos} y \texttt{blue\_master\_pos}: posiciones cacheadas de los maestros.
\end{itemize}

Las posiciones de los maestros se almacenan explícitamente para evitar recorrer el tablero cada vez que se necesita conocer su ubicación. Esta decisión simplifica y acelera operaciones frecuentes, como la comprobación de victoria o la evaluación heurística de posiciones durante la búsqueda de la inteligencia artificial.

Los elementos básicos del juego se modelan mediante tipos sencillos y separados:

\begin{itemize}
    \item \texttt{Player}, \texttt{PieceType} y \texttt{Piece}, definidos en \texttt{pieces.py}.
    \item \texttt{Card}, definida en \texttt{cards.py}, junto con el conjunto de cartas oficiales.
    \item \texttt{Move} y \texttt{Pass}, definidos en \texttt{moves.py}.
\end{itemize}

El pase se representa como una acción propia mediante \texttt{Pass}. De esta forma, incluso cuando un jugador no puede desplazar ninguna pieza, el turno sigue pasando por el mismo flujo de aplicación de reglas: se elige una carta, se intercambia con la carta lateral y cambia el jugador activo.

Las cartas se definen internamente desde la perspectiva del jugador rojo. Para obtener sus movimientos desde la perspectiva del jugador azul, el método \texttt{Card.deltas\_for(player)} rota los desplazamientos 180 grados. Además, \texttt{rules.py} precalcula una tabla de movimientos por jugador, carta y casilla, evitando recalcular orientación y límites del tablero en cada llamada a la generación de acciones.

\subsection{Aplicación de reglas}
\label{subsec:aplicacion-reglas}

La aplicación de reglas se centraliza en \texttt{rules.py}. El flujo principal del motor se organiza alrededor de las siguientes operaciones:

\begin{enumerate}
    \item A partir de un \texttt{GameState}, la función \texttt{generate\_legal\_actions(state)} obtiene las acciones disponibles para el jugador activo. Para ello, recorre sus piezas, consulta los destinos permitidos por sus dos cartas y descarta los movimientos que salen del tablero o terminan sobre una pieza propia.

    \item Si no existe ningún desplazamiento legal, se generan las dos acciones de pase posibles, \texttt{Pass(0)} y \texttt{Pass(1)}. Esto permite mantener el pase dentro del mismo modelo de acciones que los movimientos ordinarios.

    \item Una acción seleccionada se aplica mediante \texttt{apply\_action(state, action)}, que devuelve siempre un nuevo \texttt{GameState}. En una acción \texttt{Move}, se actualiza el tablero, se resuelven posibles capturas, se actualizan las posiciones cacheadas de los maestros si procede, se intercambia la carta utilizada con la carta lateral y se cambia el turno. En una acción \texttt{Pass}, el tablero no cambia, pero se realiza igualmente el intercambio de cartas y el cambio de jugador.

    \item Tras cada transición, las funciones \texttt{winner(state)} e \texttt{is\_terminal(state)} permiten comprobar si la partida ha finalizado por alguna de las condiciones de victoria.
\end{enumerate}

Esta organización hace que el motor funcione como una fuente única de verdad para las reglas del juego. La inteligencia artificial utiliza sus sucesores legales para buscar jugadas, la sesión los emplea para validar movimientos observados en el tablero físico y las interfaces se apoyan en el mismo estado para mostrar la evolución de la partida.

\section{Agente de inteligencia artificial}
\label{sec:agente-ia}

El agente de inteligencia artificial se implementa en el paquete \texttt{onitama.ai}. La implementación se reparte en varios módulos principales:

\begin{itemize}
    \item \texttt{agent.py}: selección de acciones mediante \texttt{choose\_action()}.
    \item \texttt{search.py}: búsqueda adversarial y optimizaciones.
    \item \texttt{evaluate.py}: funciones heurísticas de evaluación.
    \item \texttt{profiles.py}: perfiles de dificultad del agente.
    \item \texttt{controllers.py}: adaptación del agente mediante \texttt{AIController}.
\end{itemize}

\subsection{Selección de acciones}
\label{subsec:seleccion-acciones-ia}

El punto de entrada del agente es \texttt{choose\_action()}. Esta función recibe un \texttt{GameState} y devuelve una acción legal, de tipo \texttt{Move} o \texttt{Pass}. Su flujo principal es el siguiente:

\begin{enumerate}
    \item Comprueba si el estado recibido es terminal mediante \texttt{is\_terminal()}.
    \item Obtiene las acciones legales con \texttt{generate\_legal\_actions()}.
    \item Fija como perspectiva de evaluación el jugador que debe mover.
    \item Ordena las acciones iniciales, priorizando victorias inmediatas y capturas.
    \item Aplica cada acción con \texttt{apply\_action()} para construir los estados sucesores.
    \item Evalúa los sucesores mediante búsqueda y devuelve la acción mejor valorada.
\end{enumerate}

La búsqueda trabaja siempre sobre sucesores creados con \texttt{apply\_action()}, por lo que cada nodo explorado corresponde a un estado legal generado por el motor.

La clase \texttt{AIController} adapta el agente al protocolo común de controladores del sistema. Su configuración incluye la profundidad de búsqueda, el nombre del evaluador y la profundidad quiescente. Durante la inicialización obtiene el evaluador correspondiente mediante \texttt{get\_evaluator()} y puede mantener una tabla de transposición propia cuando esta técnica está habilitada. En cada llamada a \texttt{select\_action()}, delega la selección en \texttt{choose\_action()}.

\subsection{Búsqueda y evaluación}
\label{subsec:busqueda-evaluacion-ia}

La búsqueda se implementa en \texttt{search.py} mediante \texttt{alphabeta()}, siguiendo una formulación \textit{negamax} con poda alfa-beta. La función recibe el estado, la profundidad restante, los valores \texttt{alpha} y \texttt{beta}, la perspectiva del jugador raíz, el evaluador y las estructuras auxiliares de búsqueda.

\subsubsection{Optimización de la búsqueda}

La implementación incorpora varias técnicas configurables asociadas al proceso de búsqueda:
\begin{itemize}
    \item \textbf{Tabla de transposición}: definida como \texttt{dict[GameState, TTEntry]}. Cada entrada almacena profundidad, valor, tipo de cota y mejor acción encontrada.
    \item \textbf{Ordenación de acciones}: combina la mejor acción almacenada en la tabla de transposición, capturas, \textit{killer moves} e historial de movimientos.
    \item \textbf{Profundización iterativa}: \texttt{choose\_action()} puede lanzar búsquedas de profundidad creciente hasta alcanzar el límite configurado.
    \item \textbf{Ventanas de aspiración}: a partir de la puntuación obtenida en una profundidad anterior, se intenta primero una búsqueda con una ventana más estrecha.
    \item \textbf{Búsqueda quiescente}: cuando se alcanza la frontera de búsqueda, \texttt{quiescence()} puede extender el análisis únicamente sobre acciones de captura para evitar evaluaciones prematuras en posiciones tácticamente inestables.
\end{itemize}

Estas opciones se controlan desde los parámetros de \texttt{choose\_action()}, que permiten fijar la profundidad principal, seleccionar el evaluador y activar o ajustar las técnicas auxiliares de búsqueda. De este modo, la misma implementación puede ejecutarse con distintas configuraciones, facilitando su análisis experimental posterior.

La búsqueda determina qué estados se alcanzan durante la exploración, mientras que la función de evaluación determina cómo se valoran los estados situados en la frontera del árbol. Por ello, el evaluador utilizado influye directamente en la acción finalmente seleccionada por el agente.

\subsubsection{Funciones de evaluación}

Las funciones de evaluación se definen en \texttt{evaluate.py} y comparten una misma interfaz: reciben un \texttt{GameState} y un jugador de referencia, y devuelven una puntuación entera desde esa perspectiva. Esta interfaz permite ejecutar el algoritmo de búsqueda con distintos evaluadores sin modificar su implementación.

Todos los evaluadores tratan de forma especial los estados terminales: si la posición ya está ganada o perdida, se devuelve un valor absoluto mediante \texttt{WIN\_SCORE}, evitando que cualquier combinación de factores heurísticos tenga prioridad sobre una victoria o derrota real.

En los estados no terminales, las puntuaciones se organizan por escalas: las situaciones decisivas, como las amenazas directas sobre el maestro, reciben valores muy superiores a los criterios posicionales, como movilidad, centralidad o avance. Esta separación evita que pequeñas ventajas estratégicas compensen posiciones tácticamente peligrosas.

Sobre esta base, se implementan las siguientes versiones:

\begin{description}
    \item[\texttt{evaluate\_v1()}] Constituye la primera versión de la heurística y actúa como referencia inicial. Su evaluación se centra en factores básicos y tácticos:
    \begin{itemize}
        \item diferencia material entre estudiantes propios y rivales;
        \item progreso relativo del maestro hacia el templo contrario, aproximado mediante la distancia \textit{Manhattan};
        \item penalización de amenazas directas sobre el maestro propio;
        \item penalización de estudiantes propios situados en casillas atacadas por el rival.
    \end{itemize}

    El progreso del maestro aproxima la cercanía a una de las condiciones de victoria, mientras que las penalizaciones tácticas evitan valorar positivamente posiciones en las que las piezas propias quedan expuestas.

    \item[\texttt{evaluate\_v2()}] Extiende la versión anterior mediante una estructura de pesos configurable, \texttt{EvalWeightsV2}. Esta organización permite ajustar la importancia relativa de cada criterio sin modificar la lógica general de la evaluación.

    Además de los factores considerados en \texttt{evaluate\_v1()}, esta versión incorpora:
    \begin{itemize}
        \item valoración ofensiva de amenazas sobre el maestro rival;
        \item penalización y recompensa de estudiantes amenazados en ambos sentidos;
        \item movilidad, medida a partir de la diferencia de movimientos legales disponibles;
        \item centralidad de los estudiantes en el tablero;
        \item avance de las piezas hacia el campo contrario.
    \end{itemize}

    Con estos cambios, la evaluación incorpora información sobre presión ofensiva, capacidad de acción y ocupación posicional del tablero, además de los criterios tácticos y materiales de la versión inicial.

    \item[\texttt{evaluate\_v3()}] Introduce una evaluación dependiente de la fase de partida. Para ello, estima un factor de final de partida a partir del número total de estudiantes restantes:

    \begin{center}
    \texttt{total\_students = my\_students + opp\_students}

    \texttt{endgame = 1.0 - (total\_students / 8.0)}
    \end{center}

    Cuando permanecen los ocho estudiantes sobre el tablero, el valor de \texttt{endgame} es cero; a medida que se capturan estudiantes, el valor aumenta y la evaluación se aproxima a una fase final.

    Esta versión mantiene la valoración material y las amenazas sobre maestros y estudiantes, pero ajusta parte de sus criterios según la fase de partida:
    \begin{itemize}
        \item el progreso del maestro gana importancia en fases finales;
        \item la movilidad tiene más peso en fases iniciales y menos en finales;
        \item los criterios explícitos de centralidad y avance se sustituyen por tablas posicionales;
    \end{itemize}

    Las tablas posicionales permiten valorar la ubicación de las piezas según su tipo y su posición relativa en el tablero. En el caso de los estudiantes, favorecen el avance y la ocupación de zonas centrales; en el caso del maestro, aplican una valoración más prudente, evitando empujarlo de forma excesivamente agresiva en fases tempranas.
\end{description}

El ajuste de estos criterios y la comparación experimental entre las distintas versiones se abordan en el capítulo de experimentación.

\section{Visión por computador}
\label{sec:vision-computador}

La implementación del módulo de visión se encuentra en el paquete \texttt{onitama.vision}. Su función dentro del código es transformar un frame de cámara en una observación estructurada del tablero y de las cartas, sin decidir todavía si dicha observación corresponde a una jugada legal.

\subsection{Pipeline visual}
\label{subsec:pipeline-visual}

El punto de entrada principal es la clase \texttt{VisionPipeline}, definida en \texttt{vision\_pipeline.py}. Su método \texttt{snapshot\_from\_frame(frame)} coordina el procesamiento completo de una imagen capturada por la cámara.

El flujo implementado se apoya en dos ficheros de calibración:

\begin{itemize}
    \item \texttt{data/vision/calibration.json}: almacena los puntos del tablero, el tamaño de salida de la vista rectificada y la rotación aplicada.
    \item \texttt{data/vision/card\_rois.json}: almacena las cinco regiones de interés correspondientes a las cartas visibles.
\end{itemize}

A partir de estos datos, el pipeline procesa el frame en dos ramas: una para el tablero y otra para las cartas. La primera rectifica la zona del tablero y detecta las piezas; la segunda extrae las regiones de cartas y clasifica cada una de ellas. Finalmente, ambas salidas se combinan en un \texttt{VisionSnapshot}.

\subsection{Detección de piezas y clasificación de cartas}
\label{subsec:deteccion-piezas-clasificacion-cartas}

El módulo utiliza estrategias distintas para piezas y cartas porque la información que debe recuperarse en cada caso no es la misma. Las piezas deben identificarse y localizarse sobre el tablero para asignarlas a una casilla concreta, por lo que se emplea detección de objetos. En cambio, las cartas ocupan posiciones fijas en el montaje y se encuentran delimitadas mediante ROIs previamente calibradas, de modo que el problema se reduce a clasificar cada recorte para reconocer su identidad.

\subsubsection{Detección de piezas}

El reconocimiento de piezas parte de la calibración del tablero implementada en \texttt{homography.py}. A partir de \texttt{data/vision/calibration.json}, se cargan los cuatro puntos del tablero, el tamaño de salida y la rotación necesaria para obtener una vista rectificada. En la detección de piezas, esta rectificación se genera con un margen adicional alrededor del tablero, lo que permite conservar piezas que sobresalen parcialmente del área estricta de juego.

Sobre la imagen rectificada actúa \texttt{YoloPieceDetector}, definido en \\ \texttt{piece\_detector.py}. El modelo utilizado se carga desde:

\begin{center}
\texttt{models/pieces\_yolo11s\_640\_best.pt}
\end{center}

Después de la detección, cada caja se asocia a una celda \texttt{(row, col)} mediante un punto de anclaje calculado sobre la propia detección. Las clases reconocidas por el detector son:

\begin{center}
\texttt{red\_master, red\_student, blue\_master, blue\_student}
\end{center}

Cada detección se representa mediante \texttt{PieceDetection}, que almacena la clase predicha, la confianza, la caja delimitadora, el punto de anclaje y la celda asociada. El detector utiliza un umbral de confianza de \texttt{0.50} al ejecutar el modelo YOLO. Las predicciones que no alcanzan este valor no son devueltas por el modelo y, por tanto, no llegan a convertirse en objetos \texttt{PieceDetection}.

Finalmente, \texttt{detections\_to\_board()} construye un \texttt{VisionBoard}. Durante esta conversión se descartan las detecciones que no caen dentro de ninguna celda del tablero o cuya clase no puede transformarse en una pieza visual válida. Si varias detecciones se asignan a la misma casilla, se conserva únicamente la de mayor confianza.

\subsubsection{Clasificación de cartas}

En el caso de las cartas, las regiones de interés se gestionan en \texttt{card\_rois.py}. Estas regiones se cargan desde \texttt{data/vision/card\_rois.json} y corresponden a cinco posiciones fijas:

\begin{center}
\texttt{red\_0, red\_1, side, blue\_0, blue\_1}
\end{center}

Cada ROI se recorta del frame original, se enmascara si es necesario y se prepara como una imagen cuadrada antes de enviarla al clasificador. La clasificación se realiza mediante \texttt{YoloCardClassifier}, definido en \texttt{card\_classifier.py}, utilizando el modelo:

\begin{center}
\texttt{models/cards\_yolo11n-cls\_320\_best.pt}
\end{center}

Este modelo clasifica cada recorte como una de las 16 cartas oficiales del juego base. La predicción individual se representa con \texttt{CardSlotPrediction}, que incluye el slot, la clase predicha y la confianza. El clasificador aplica un umbral mínimo de confianza de \texttt{0.50} sobre la clase más probable de cada recorte. Si la confianza es inferior a este valor, se genera un error de observación de tipo \texttt{LOW\_CONFIDENCE\_CARD} y no se construye el resultado completo de clasificación.

El resultado conjunto se agrupa en \texttt{CardClassificationResult}, desde donde se recuperan las dos cartas rojas, las dos cartas azules y la carta lateral.


\subsection{Snapshot visual}
\label{subsec:snapshot-visual}

La salida principal del módulo de visión es \texttt{VisionSnapshot}, definido en \texttt{snapshot.py}. Esta estructura combina:

\begin{itemize}
    \item \texttt{board}: objeto \texttt{VisionBoard} con la disposición detectada de las piezas.
    \item \texttt{red\_cards}: nombres de las dos cartas visibles del jugador rojo.
    \item \texttt{blue\_cards}: nombres de las dos cartas visibles del jugador azul.
    \item \texttt{side\_card}: nombre de la carta lateral.
\end{itemize}

El snapshot representa una observación visual completa, pero todavía no contiene toda la información necesaria para ser un estado lógico de partida. En concreto, la cámara puede detectar piezas y cartas, pero no determina por sí misma a qué jugador le corresponde mover. Por ese motivo, la conversión mediante \texttt{to\_game\_state(to\_move)} recibe el turno como parámetro externo.

Durante esta conversión se transforman las piezas visuales en piezas del motor, se resuelven los nombres de cartas mediante \texttt{CARD\_BY\_NAME} y se aplican validaciones básicas: límites de piezas por jugador, existencia de como máximo un maestro por color y unicidad de las cinco cartas visibles. Si estas comprobaciones se cumplen, se construye un \texttt{GameState} compatible con el motor.

Esta separación permite mantener dos niveles distintos: 
\begin{itemize}
    \item \texttt{VisionSnapshot} como representación de lo observado en la imagen, con información visual y posibles inconsistencias.
    \item \texttt{GameState} como representación lógica de una posición de partida, que solo se construye si la observación es coherente.
\end{itemize}

% 5.5 Integración de componentes
% =============================================================================
\section{Integración de componentes}
\label{sec:integracion-componentes}

La integración se organiza en dos niveles. La sesión mantiene la coherencia entre las observaciones estabilizadas y el estado lógico de la partida, mientras que el control de ejecución coordina cámara, visión, sesión e interfaz durante la ejecución en vivo.

\subsection{Sesión y sincronización}
\label{subsec:sesion-sincronizacion}

La capa de sesión se implementa en el paquete \texttt{onitama.session} y trabaja sobre objetos \texttt{GameState} ya construidos a partir de la observación visual. 

Su implementación se reparte en tres módulos: \texttt{stabilizer.py}, encargado de estabilizar observaciones repetidas; \texttt{synchronizer.py}, encargado de compararlas con estados legales del motor; y \texttt{vision\_session.py}, donde se coordina la evolución de la partida asistida por visión.

\subsubsection{Estabilización de observaciones}

El módulo \texttt{stabilizer.py} define \texttt{StateStabilizer}, encargado de filtrar observaciones visuales inestables. Para ello, recibe estados observados mediante \texttt{push(observed\_state)} y mantiene internamente un estado candidato junto con el número de veces que se ha repetido de forma consecutiva.

Si la observación cambia, el candidato se sustituye y el contador vuelve a empezar. Si se repite, el contador aumenta. Cuando se alcanza el umbral \texttt{required\_repeats}, cuyo valor por defecto en la aplicación es \texttt{3}, el estado se considera estable y puede pasar a la fase de sincronización. Este mecanismo reduce el efecto de lecturas aisladas, oclusiones breves o movimientos físicos todavía incompletos.

\subsubsection{Validación de sucesores legales}

El módulo \texttt{synchronizer.py} se encarga de comprobar si una observación estable puede aceptarse como evolución válida de la partida. La función principal es \texttt{match\_observed\_state(previous\_state, observed\_state)}, que compara el estado observado con el estado confirmado y devuelve un resultado de sincronización.

El criterio aplicado es el siguiente:

\begin{center}
\begin{tabular}{ll}
\texttt{observed\_state == previous\_state} & \texttt{UNCHANGED} \\
\texttt{observed\_state} es sucesor legal & \texttt{ACCEPTED} \\
ninguna de las anteriores & \texttt{REJECTED}
\end{tabular}
\end{center}

Para comprobar el caso de aceptación, se generan las posiciones alcanzables desde \texttt{previous\_state} mediante \texttt{legal\_successors(previous\_state)}, que internamente utiliza \texttt{generate\_legal\_actions()} y \texttt{apply\_action()} del motor.

\subsubsection{Coordinación de la partida asistida}

La coordinación de la partida asistida se implementa en \texttt{VisionGameSession}, definida en \texttt{vision\_session.py}. Esta clase concentra el estado de la sesión y decide cómo debe interpretarse cada observación estable según la fase actual de la partida.

Los elementos principales que mantiene son:

\begin{itemize}
    \item \texttt{current\_state}: último estado confirmado de la partida.
    \item \texttt{phase}: fase actual de la sesión.
    \item \texttt{human\_player} y \texttt{ai\_player}: jugadores asignados al humano y a la IA.
    \item \texttt{ai\_controller}: controlador encargado de seleccionar la acción de la IA.
    \item \texttt{stabilizer}: filtro temporal de observaciones.
    \item \texttt{last\_ai\_action}: última acción seleccionada por la IA, pendiente de ejecución física.
    \item \texttt{expected\_state}: estado que debería observarse tras ejecutar físicamente la acción de la IA.
\end{itemize}

El método principal es \texttt{process\_observation(observed\_state)}. Cada observación recibida se pasa primero por el estabilizador; si todavía no se ha repetido lo suficiente, la sesión continúa esperando. Cuando se obtiene un estado estable, el comportamiento depende de la fase actual:

\begin{description}
    \item[\texttt{BOOTSTRAP}] Inicializa la sesión. El primer estado estable se adopta como \texttt{current\_state} y, a partir de él, se determina si debe mover el jugador humano, la inteligencia artificial o si la partida ya está finalizada.

    \item[\texttt{WAITING\_HUMAN\_MOVE}] Espera una jugada física del jugador humano. El estado observado se compara con \texttt{current\_state} y con sus sucesores legales. Si coincide con un sucesor, se acepta como nuevo estado confirmado; si no, se rechaza sin modificar la partida.

    \item[\texttt{READY\_FOR\_AI}] Solicita una acción al \texttt{AIController} mediante \texttt{run\_ai\_turn()}. A partir de la acción seleccionada, calcula el estado esperado, almacenado en \texttt{expected\_state}, y guarda la acción en \texttt{last\_ai\_action} para comprobar posteriormente su ejecución física.

    \item[\texttt{WAITING\_AI\_EXECUTION}] Espera la ejecución física de la jugada indicada por la IA. Si la observación coincide con \texttt{expected\_state}, se confirma la jugada; si coincide con \texttt{current\_state}, la sesión sigue esperando; si no coincide con ninguno de los dos, se considera una discrepancia de ejecución.

    \item[\texttt{FINISHED}] La partida ha terminado. El estado final queda fijado y no se aceptan nuevas transiciones.
\end{description}


\subsection{Control de ejecución}
\label{subsec:control-ejecucion}

El control de ejecución se implementa en el paquete \texttt{onitama.runtime}. Esta capa conecta la cámara, el pipeline de visión, la conversión a \texttt{GameState} y la sesión de juego, descrita anteriormente, generando un estado preparado para ser consumido por la interfaz gráfica o por herramientas de terminal.

\subsubsection{Configuración y estado expuesto}

La configuración del runtime se representa mediante \texttt{VisionRuntimeConfig}. Esta estructura agrupa los parámetros necesarios para iniciar una partida asistida por visión:

\begin{itemize}
    \item Jugador humano seleccionado.
    \item Número de repeticiones necesarias para estabilizar una observación.
    \item Evaluador, profundidad de búsqueda y profundidad de búsqueda quiescente obtenidos a partir del perfil de dificultad seleccionado.
    \item Dispositivo de cámara, resolución y tasa de frames.
\end{itemize}

A partir de esta configuración, el runtime construye internamente los componentes necesarios: cámara, \texttt{VisionPipeline}, \texttt{VisionGameSession} y \texttt{AIController}.

El estado expuesto hacia las interfaces se representa con \texttt{VisionRuntimeState}. Esta estructura contiene la información que la GUI o la CLI necesitan mostrar sin acceder directamente a los detalles internos de sesión o visión: si el runtime está activo, la fase actual, el último resultado producido, el estado confirmado, la jugada pendiente de la IA, posibles errores y el ganador si la partida ha terminado.

\subsubsection{Bucle de ejecución}

La clase principal es \texttt{VisionGameRuntime}. Sus métodos básicos son:

\begin{itemize}
    \item \texttt{start()}: abre la cámara y deja el runtime preparado para procesar frames.
    \item \texttt{stop()}: libera la cámara y detiene la ejecución.
    \item \texttt{reset()}: reconstruye la sesión y limpia errores, avisos y resultados anteriores.
    \item \texttt{step()}: ejecuta una iteración del sistema.
\end{itemize}

El método más importante es \texttt{step()}, ya que concentra la coordinación entre componentes. Su comportamiento depende del estado de la sesión:

\begin{enumerate}
    \item Si el runtime no está activo, devuelve el estado actual sin procesar nuevos datos.
    \item Si la partida ha terminado, no lee más frames de cámara.
    \item Si la sesión está en fase \texttt{READY\_FOR\_AI}, ejecuta \texttt{run\_ai\_turn()} sin consultar la cámara.
    \item En el resto de fases, lee un frame, obtiene un \texttt{VisionSnapshot} mediante \texttt{snapshot\_from\_frame()}, lo convierte a \texttt{GameState} y entrega el estado observado a \texttt{process\_observation()}.
\end{enumerate}

La conversión de \texttt{VisionSnapshot} a \texttt{GameState} depende de la fase de la sesión. Durante el arranque, el jugador inicial se obtiene a partir del sello de la carta lateral. Durante la partida, el runtime interpreta la observación usando el turno del estado confirmado o del estado esperado, según corresponda. Esta lógica es necesaria porque el módulo de visión reconoce piezas y cartas, pero no controla por sí mismo el turno actual de la partida.

Además, el runtime centraliza los errores y avisos que pueden aparecer durante la ejecución, como problemas de cámara, modelos o calibración, así como observaciones visuales inválidas. Estos datos se incorporan a \texttt{VisionRuntimeState} para que la interfaz pueda informar al usuario sin acceder directamente a las capas internas.

\section{Interfaz gráfica}
\label{sec:interfaz-grafica}

La interfaz gráfica se implementa en el paquete \texttt{onitama.gui} mediante PySide6. Esta capa consume el estado proporcionado por el runtime y lo traduce en elementos visuales e interacciones de usuario.

\subsection{Estructura de la aplicación}
\label{subsec:estructura-gui}

El punto de entrada de la aplicación es \texttt{vision\_app.py}, asociado al ejecutable \texttt{onitama-vision}. Este módulo crea la aplicación Qt, instancia \texttt{MainWindow} y muestra la ventana principal.

La ventana principal se implementa en \texttt{main\_window.py}. Su estructura se basa en un \texttt{QStackedWidget}, que permite alternar entre la pantalla inicial de configuración y la vista de partida asistida por visión. Cuando se inicia una partida, \texttt{MainWindow} valida la configuración, construye el \texttt{VisionRuntimeConfig}, cambia a la vista de juego y arranca la ejecución mediante un worker en segundo plano.

Los componentes principales de la interfaz se reparten de la siguiente forma:

\begin{itemize}
    \item \texttt{setup\_page.py}: pantalla inicial de configuración.
    \item \texttt{runtime\_worker.py}: adaptación del runtime al modelo de hilos y señales de Qt.
    \item \texttt{widgets.py}: widgets personalizados para tablero, cartas y mensajes.
    \item \texttt{view\_logic.py}: traducción del estado del runtime a mensajes de interfaz.
    \item \texttt{camera\_window.py}: ventana auxiliar para visualizar el último frame de cámara.
    \item \texttt{gui/calibration/}: diálogos integrados de calibración del tablero y de las cartas.
\end{itemize}

Esta organización permite que \texttt{MainWindow} actúe como coordinador de la interfaz, mientras que la lógica de ejecución, la representación visual, los mensajes y la calibración se mantienen en componentes separados.

\subsection{Configuración y calibración}
\label{subsec:configuracion-calibracion-gui}

La pantalla inicial se define en \texttt{SetupPage} y reúne las decisiones previas a la partida: el color del jugador humano, el nivel de dificultad del agente de inteligencia artificial y el estado de la calibración visual.

El color seleccionado determina qué jugador será controlado por el usuario, mientras que la dificultad se utiliza para construir la configuración del agente que se enviará al runtime. Por su parte, la calibración visual depende de dos ficheros requeridos por el módulo de visión:

\begin{itemize}
    \item \texttt{data/vision/calibration.json}: calibración geométrica del tablero.
    \item \texttt{data/vision/card\_rois.json}: regiones de interés de las cartas.
\end{itemize}

Antes de habilitar el inicio de partida, la interfaz comprueba que ambos ficheros existan y sean válidos. Si falta alguno de ellos, el usuario puede generarlo desde la propia aplicación mediante diálogos integrados en Qt.

\texttt{BoardCalibrationDialog} permite marcar las cuatro esquinas del tablero y guardar la transformación geométrica. \texttt{CardRoisCalibrationDialog} permite definir las regiones \texttt{red\_0}, \texttt{red\_1}, \texttt{side}, \texttt{blue\_0} y \texttt{blue\_1}. Ambos diálogos utilizan captura de cámara propia, permiten congelar la imagen y editar los puntos de forma interactiva antes de guardar la configuración.

\subsection{Vista de partida y actualización en vivo}
\label{subsec:vista-partida-gui}

La vista de partida muestra el estado reconstruido del juego a partir de los datos recibidos desde el runtime. El tablero se representa mediante \texttt{BoardWidget}, que dibuja la cuadrícula, los templos, las piezas y, cuando existe, la jugada pendiente de la inteligencia artificial. Las cartas se representan mediante \texttt{CardWidget}, mostrando su nombre, sello de color y patrón de movimientos. Los mensajes de estado se presentan mediante \texttt{MessageBanner}.

La ejecución en vivo no se realiza en el hilo principal de la interfaz. Para evitar bloqueos durante la captura de cámara, inferencia visual, estabilización y cálculo de la IA, la GUI utiliza \texttt{RuntimeWorker}, implementado como un \texttt{QThread}. Este worker encapsula \texttt{VisionGameRuntime} y emite señales hacia la ventana principal:

\begin{itemize}
    \item \texttt{state\_changed}: nuevo \texttt{VisionRuntimeState}.
    \item \texttt{frame\_changed}: nuevo frame convertido a \texttt{QImage}.
    \item \texttt{failed}: error producido durante la ejecución.
\end{itemize}

La ventana principal no accede directamente a \texttt{VisionGameSession} ni a \texttt{VisionPipeline}. En su lugar, recibe objetos \texttt{VisionRuntimeState} y actualiza la interfaz mediante \texttt{\_apply\_state()}. Esta estructura mantiene separada la presentación visual de la lógica interna de captura, visión y sincronización.

La traducción de fases, resultados y errores a mensajes comprensibles para el usuario se concentra en \texttt{view\_logic.py}. La función \texttt{build\_status\_view()} recibe un \texttt{VisionRuntimeState} y produce un \texttt{StatusView}, compuesto por título, detalle y tono visual. De esta forma, los mensajes mostrados por la interfaz se mantienen separados de la lógica de sesión.

La aplicación también puede abrir una ventana secundaria de cámara mediante \texttt{CameraWindow}. Esta vista permite observar el último frame capturado sin mezclar la representación de cámara con la vista principal de partida.

\section{Interfaces auxiliares y herramientas de desarrollo}
\label{sec:interfaces-herramientas}

Además de la aplicación gráfica principal, el proyecto incluye interfaces y herramientas auxiliares destinadas a ejecutar, depurar y evaluar componentes de forma independiente.

El paquete \texttt{onitama.cli} proporciona interfaces de terminal para dos escenarios principales:

\begin{itemize}
    \item \texttt{play.py}: ejecución de partidas locales sin visión, con modos humano contra humano, humano contra IA, IA contra humano e IA contra IA.
    \item \texttt{vision\_play.py}: ejecución por terminal del flujo asistido por visión, utilizando \texttt{VisionGameRuntime}.
\end{itemize}

Estas interfaces permiten probar el motor, la inteligencia artificial y la integración con visión sin depender necesariamente de la interfaz gráfica.

La carpeta \texttt{scripts} reúne herramientas de apoyo al desarrollo. Entre ellas se incluyen scripts de calibración, previsualización, captura de datos, depuración del pipeline visual, construcción de snapshots, torneos entre agentes y medición del rendimiento de la búsqueda. Estas utilidades no forman parte del flujo principal de uso, pero han sido importantes para ajustar y validar los distintos módulos durante el desarrollo.

Por último, la carpeta \texttt{tests} contiene pruebas automatizadas ejecutadas con \texttt{pytest}. Estas pruebas cubren aspectos como las reglas del motor, la aplicación de acciones, las condiciones de victoria, la conversión entre visión y motor, la estabilización de observaciones, la sincronización y partes del runtime.

En conjunto, estas interfaces y herramientas han servido como soporte para construir el sistema de forma incremental, facilitar la detección de errores y disponer de una base práctica para la evaluación experimental que se aborda en el capítulo siguiente.
